Ciclo de vida y arquitectura MVC

Ciclo de vida — Application

JavaFX
public class ViMapApp extends Application {
  @Override
  public void init() throws Exception {
    // Antes de UI: configs, BD, servicios
    super.init();
  }
  @Override
  public void start(Stage stage) {
    // Punto de entrada de la UI
    var loader = new FXMLLoader(
        getClass().getResource("/fxml/main.fxml"));
    Parent root = loader.load();
    Scene scene = new Scene(root, 1280, 800);
    stage.setScene(scene);
    stage.show();
  }
  @Override
  public void stop() throws Exception {
    // Limpieza: BD, estado
  }
}

    new FilteredList<>(masterList, p -> true);

    filtered.setPredicate(p ->

    new SortedList<>(filtered);

Task — operaciones en segundo plano

threading
Task<List<MapPoint>> task = new Task<>() {
  @Override
  protected List<MapPoint> call() throws Exception {
    updateMessage("Cargando...");
    updateProgress(i+1, total);
    return repo.findAll();
  }
};
progressBar.progressProperty().bind(task.progressProperty());
task.setOnSucceeded(e -> table.setItems(...));
task.setOnFailed(e -> showError(...));
new Thread(task).start();
⚠ Regla de oroNunca actualices nodos de UI desde otro hilo. Usa Platform.runLater().

Repository + Service Layer

patrón
// Repository — acceso a datos
public MapPoint save(MapPoint p) throws SQLException {
  try (Connection conn = DatabaseConfig.getConnection();
       PreparedStatement ps = conn.prepareStatement(INSERT)) {
    ps.setString(1, p.getName());
    ps.setDouble(2, p.getLatitude());
    conn.commit();
  }
}

// Service — carga async en ObservableList
public Task<Void> loadAllAsync() {
  return new Task<>() {
    protected Void call() throws Exception {
      List<MapPoint> pts = repo.findAll();
      Platform.runLater(() -> points.setAll(pts));
      return null;
    }
  };
}

Buenas prácticas — Checklist

Checklist de calidad ViMap

producción
✓UI en hilo correctoToda actualización de UI va en Platform.runLater() o callbacks de Task
✓Sin lógica en FXMLLos Controllers solo orquestan; la lógica va en Services
✓Bindings reactivosUsar propiedades JavaFX en lugar de setters manuales
✓CSS externalizadoNingún setStyle() inline en producción
✓Pool de conexionesUsar HikariCP, nunca crear conexiones directas
✓Manejo de erroresTask.setOnFailed() siempre definido
✓Resources cerradostry-with-resources en todo acceso JDBC
✓Tests unitariosServices y Repositories testeados con JUnit 5
✗Memory leaksDesregistrar listeners al destruir el controller o usar WeakChangeListener
✗Bloquear hilo UINunca usar Thread.sleep() o IO dentro de un EventHandler
✗Nodos UI en TaskConstruir datos en Task, crear nodos en Platform.runLater()
✗Credenciales en códigoUsar variables de entorno o archivo externo en .gitignore
